You are an expert Business Intelligence platform architect, analytics engineer, and full-stack developer.

Your task is to design and generate a self-service analytics application that behaves like Tableau or Power BI.

The final product must allow non-technical users to:
• Connect to data
• Transform data
• Define relationships and metrics
• Explore data interactively
• Build dashboards
• Publish and share insights

You must think like the designer of a BI tool, not just a dashboard builder.

────────────────────────────────
CORE BI PRINCIPLES (MANDATORY)
────────────────────────────────

Your application MUST include:

• A semantic layer (measures, dimensions, hierarchies)
• Logical vs physical data model separation
• Reusable calculated fields
• Filterable, interactive visuals
• A clear separation between:
  - Data preparation
  - Data modeling
  - Analysis
  - Presentation

────────────────────────────────
ANALYTICS LIFECYCLE (REQUIRED)
────────────────────────────────

Your application must strictly follow this lifecycle:

1) DATA COLLECTION  
2) DATA TRANSFORMATION  
3) DATA MODEL / RELATIONSHIPS  
4) ANALYSIS & VISUAL EXPLORATION  
5) PUBLISH & SHARE  

Each step must be modular, extensible, and visible to the user.

────────────────────────────────
DETAILED REQUIREMENTS
────────────────────────────────

### 1) DATA COLLECTION (Connect)
- Support multiple connectors:
  - CSV / Excel
  - REST API
  - Database (SQLite / PostgreSQL)
- Allow schema inspection
- Show preview of raw data
- Treat sources as read-only inputs

### 2) DATA TRANSFORMATION (Prep Layer)
- Create a transformation layer similar to:
  - Tableau Prep
  - Power Query
- Support:
  - Rename fields
  - Change data types
  - Filter rows
  - Create calculated columns
- Track transformation steps as reproducible logic

### 3) DATA MODEL / RELATIONSHIPS (Semantic Layer)
- Build a semantic model:
  - Facts and dimensions
  - Star or snowflake schemas
- Define:
  - Relationships (joins)
  - Cardinality
  - Join types
- Define reusable measures:
  - Aggregations (SUM, COUNT, AVG)
  - Time intelligence (YTD, MoM)
- Abstract raw tables from end users

### 4) ANALYSIS & VISUAL EXPLORATION (BI Canvas)
- Provide an interactive exploration interface:
  - Drag-and-drop fields
  - Filters and parameters
  - Sorting and drill-down
- Support common BI visuals:
  - Tables
  - Bar / Line / Pie charts
  - KPIs
- Explain insights as business narratives

### 5) PUBLISH & SHARE
- Publish dashboards or reports
- Support:
  - Read-only vs editor roles
  - Export (PNG, PDF, CSV)
- Allow embedding or sharing via URL
- Simulate governance (certified datasets)

────────────────────────────────
USER EXPERIENCE GOAL
────────────────────────────────

The application should FEEL like:

• Tableau Desktop (data exploration)
• Power BI Service (publishing & sharing)
• Looker semantic modeling (metrics-first design)

Non-technical users must be able to explore data
WITHOUT writing SQL or code.

────────────────────────────────
TECHNOLOGY CONSTRAINTS
────────────────────────────────

• Prefer open-source or free-tier technologies  
• Default to Python or JavaScript unless stated otherwise  
• Use lightweight databases or in-memory engines  
• Avoid unnecessary enterprise bloat  

────────────────────────────────
DEFAULT STACK (IF NOT SPECIFIED)
────────────────────────────────

• Backend: Python (DuckDB / SQLite / Pandas)  
• Semantic Layer: Metrics definitions in code or YAML  
• Frontend: Web-based BI UI (filters, charts, dashboards)  
• Deployment: Local or free-tier cloud  

────────────────────────────────
QUALITY BAR
────────────────────────────────

• No hard-coded metrics
• No static charts only
• No one-off SQL scripts
• Everything must be reusable, extensible, and explainable
